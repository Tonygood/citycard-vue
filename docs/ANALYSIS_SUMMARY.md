### 4. 关键问题

**问题1：代码重复率高达30-40%**
- 疲劳减半逻辑在3处重复
- 攻击力计算在3处重复
- 伤害应用在3处重复
- 蓝色奖励有3种不同实现

**问题2：缺乏统一的数据结构**
- battle2P: `currentBattleCities[]` 数组
- battle3P: `currentBattleData{}` 对象
- 在confirmBattleAction中需要转换

**问题3：功能不完整**
- battle3P缺少游戏结束判定逻辑
- battle3P缺少中心城市阵亡淘汰处理
- 缺少潜能激发处理（3P）

**问题4：耦合度过高**
- 所有函数直接访问roomData全局对象
- 技能状态散布在gameState的20+个嵌套对象中
- 难以追踪状态变更

### 5. 技能系统复杂度

**2P模式支持的技能**: 20+个
- 欲擒故纸、以逸待劳、趁火打劫
- 背水一战、狂暴模式、中心城市翻倍
- 潜能激发、蓝色奖励、同归于尽
- 御驾亲征、等等

**3P特有**: 隔岸观火
**2v2特有**: 挑拨离间

## 性能分析

### 时间复杂度

| 操作 | 复杂度 | 备注 |
|------|--------|------|
| 一轮战斗 | O(n²) | n=出战城市数，通常n≤3 |
| 伤害结算 | O(n) | 线性扫描防守方城市 |
| 技能检查 | O(1) | HashMap查询 |
| 日志输出 | O(k) | k=日志条目数 |

### 优化机会

1. **预计算**：在战斗前计算所有城市的有效攻击力 (15-20%)
2. **缓存**：缓存技能加成结果 (5-10%)
3. **早停**：伤害为0时提前退出循环 (5-15%)
4. **索引**：使用Set替代Array存储已死亡城市 (10-20%)

**总体优化空间**: 20-30%
