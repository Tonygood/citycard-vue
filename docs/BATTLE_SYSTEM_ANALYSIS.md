## 四、战斗系统的代码特征分析

### 4.1 重复与冗余分析

#### 代码重复情况

**高度重复的代码模式**:

1. **疲劳减半处理** - 三个函数都有
   ```javascript
   // 在battle2P、battle3P、battle2v2中各出现一次
   const prevStreak = player.streaks[idx] || 0;
   if (prevStreak >= 1) {
     city.hp = Math.floor(city.hp / 2);
   }
   ```

2. **绿色技能减伤** - 在每个战斗函数中重复
   ```javascript
   if (card.green > 0) {
     reduction = getGreenBonus(card.green);
   }
   ```

3. **蓝色奖励** - 三个函数各有一套逻辑
   - battle2P: 检查对手全部阵亡
   - battle3P: 检查每对玩家的摧毁关系
   - battle2v2: 检查整队是否全灭

4. **背水一战后置处理** - 相似但略有不同
   - battle2P和battle2v2都有，but处理范围不同

5. **中心城市阵亡淘汰** - battle2P和battle2v2各有一套
   - battle3P缺失此处理

6. **攻击力计算** - 都包含
   - 红色技能加成
   - 中心城市翻倍（仅2P/2v2）
   - 御驾亲征加成
   - 背水一战翻倍
   - 狂暴模式5倍

**重复代码估计**: **30-40%** 的代码存在不同程度的重复

#### 不一致之处

1. **battle3P缺少以下逻辑**:
   - 中心城市阵亡淘汰处理
   - 游戏结束判定
   - 潜能激发处理

2. **battle2v2特有逻辑**:
   - 挑拨离间（可能导致提前返回）
   - 队伍维度的聚合处理
   - 潜能激发伤害分配给对方队员

3. **日志输出差异**:
   - battle2P: 使用`addPublicLog`
   - 新的云存储版本在helpers.js中定义

### 4.3 模块化难度评估

#### 当前状况评估

| 方面 | 评分 | 说明 |
|------|------|------|
| **耦合度** | 高 | 三个战斗函数都直接访问roomData全局对象 |
| **参数化** | 低 | 大量硬编码的值（如HP阈值、倍数等） |
| **错误处理** | 低 | 缺乏验证和错误提示 |
| **可测试性** | 低 | 无法单独测试，依赖完整的game state |
| **文档完整性** | 中 | 部分函数有JSDoc注释 |
| **配置外部化** | 无 | 所有常量硬编码 |

#### 重构难度分级

**难度等级**: ⭐⭐⭐⭐ (中高难度)

**主要挑战**:

1. **状态依赖复杂**
   - gameState包含20+个嵌套对象
   - 技能状态散布各处
   - 难以追踪状态变更

2. **代码顺序敏感**
   - 后置处理必须在伤害结算之后
   - 技能处理有先后依赖
   - 修改一处可能影响多处

3. **两种架构风格混用**
   - 旧的全局state对象
   - 新的roomData对象
   - 未完全迁移

4. **分支条件众多**
   - 20+个特殊技能的条件判断
   - 模式判定（2P/3P/2v2）
   - 状态检查分散

#### 推荐的重构策略

**分阶段方案**:

**第1阶段** (低风险，高收益):
- 提取纯函数：`applyFatigue()`、`calculateAttackPower()`
- 创建常量配置文件：SKILL_CONFIGS
- 为重复代码添加测试

**第2阶段** (中等难度):
- 创建战斗事件系统（发射器模式）
- 提取技能效果处理器
- 创建伤害管理系统

**第3阶段** (高难度):
- 战斗状态机重构
- 完全替换为Function Programming风格
- 从roomData迁移到专用GameState对象

**不建议一次性全部重构**:
- 风险高
- 难以验证正确性
- 容易引入新bug

## 五、关键代码片段分析

### 5.1 战斗流程控制链

```
confirmBattleAction() 
  ↓ 确认出战，保存出战数据
checkAndProcessBattle()
  ↓ 检查所有玩家是否ready
battle2P/3P/2v2()
  ↓ 执行战斗逻辑
endOfRoundTickProcessing()
  ↓ 处理倒计时
自动获得3金币
  ↓
saveRoomData()
```

### 5.2 技能处理顺序 (battle2P为例)

1. **前置处理** (战斗前)
   - 欲擒故纸检查与处理
   - 疲劳减半

2. **伤害计算** (战斗中)
   - 计算攻击力（红色加成）
   - 应用减伤（绿色技能）
   - 计算实际伤害

3. **后置处理** (战斗后)
   - 潜能激发
   - 蓝色奖励
   - 背水一战
   - 同归于尽
   - 狂暴模式禁用
   - 中心城市淘汰
   - 游戏结束判定
   - 以逸待劳抢夺金币
   - 趁火打劫抢夺金币

**关键发现**: 技能处理分为三个清晰的阶段，但目前全部内联在函数中

### 5.3 数据结构差异

**battle2P中**:
```javascript
state.currentBattleCities = [
  {cityIdx: 0, red: 2, green: 0, blue: 1},
  ...
]
```

**battle3P中**:
```javascript
state.currentBattleData = {
  "玩家B": [{cityIdx: 1, red: 1, ...}, ...],
  "玩家C": [{cityIdx: 2, red: 0, ...}, ...]
}
```

**差异影响**: 需要在confirmBattleAction中根据模式转换数据结构

## 七、性能分析

### 当前性能特点

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| 一轮战斗 | O(n²) | n=出战城市数，二重循环遍历 |
| 伤害计算 | O(n) | n=防守城市数，线性扫描 |
| 技能检查 | O(1) | 直接键值查询 |
| 日志输出 | O(k) | k=技能数量 |

### 优化机会

1. **预计算**: 提前计算所有城市的有效攻击力
2. **缓存**: 缓存技能加成计算结果
3. **早停**: 当伤害为0时提前退出循环
4. **索引**: 为已死亡城市使用Set而不是Array
